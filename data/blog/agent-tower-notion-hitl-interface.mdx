A multi-agent pipeline with four stages in flight has an observation problem. Each agent writes to a terminal. As the fleet grows there is no glass cockpit, no ambient surface where an operator can see who is doing what, catch a wrong hypothesis in stage two and let stages three and four continue.

LangGraph and similar frameworks solve the mechanism: pause, persist state, resume. What they do not solve is the interface. Every HITL demo ends with "and then you build a UI for the human."

Agent Tower skips the custom UI. The Notion kanban board your team already uses becomes the cockpit. One card per incident run. Comments are the control channel.

![Notion kanban board with four incident pipeline stages visible, one card in 'Needs Review'](/static/images/agent-tower-notion-hitl-interface/kanban-fleet.jpg)

## Architecture

Four stages: Triage, Investigation, Remediation and Comms. Each stage is a [Hermes Agent](https://github.com/NousResearch/hermes) session, configured here to run Claude. `pipeline.py` sequences the stages, updates Notion card status and implements two HITL patterns via comment polling.

![Four-stage incident pipeline with HITL pattern annotations and memory-layer callouts per stage](/static/images/agent-tower-notion-hitl-interface/pipeline-flow.svg)

**Pattern B** (Investigation): the agent posts its findings to the card as a comment, then polls for 20 seconds. If the operator leaves a comment during that window, the stage re-runs with the operator note injected. No comment: auto-advance.

**Pattern A** (Comms): the agent posts a draft and moves the card to "Needs Review," then polls indefinitely. A keyword approval ("lgtm", "approved", "ship it") sends the brief unchanged. Anything else is a redirect: the stage re-runs with the note applied.

Both patterns use the same `_poll_for_new_comment` function and the same Notion comments API. The difference is the timeout and the approval gate.

## Implementation

### Triage as a structured decision gate

Triage classifies the incident and branches the pipeline. Every Triage run emits a `handoff` YAML block that `pipeline.py` parses to decide what runs next.

Three decisions are possible: `investigate` (default), `auto-remediate` (skip Investigation when the KB match is exact and symptoms are unambiguous) and `escalate` (pause on "Needs Review" when confidence is too low to proceed).

```yaml
handoff:
  stage_completed: triage
  decision: investigate
  confidence: medium
  rationale: KB pattern matched on connection pool exhaustion; log signals consistent but a competing rollback signal warrants investigation
  notes_for_next_stage:
    - DB connection acquire latency above threshold from 14:32:09
    - config-service v2.3.1 deploy started at 14:30
    - transient packet loss WARN at 14:32:01-14:32:14
```

`pipeline.py` reads `decision` and branches:

```python
if decision == "auto-remediate":
    skip_stages.add("Investigation")
elif decision == "escalate":
    _set_status(card_id, "Needs Review")
    _post_comment(card_id, f"Triage escalated (confidence: {confidence}). Rationale: {handoff.get('rationale', 'not provided')}", label="Triage — Escalated")
    return 0
```

The Triage prompt has an explicit constraint: do not name root causes. Triage classifies symptoms and finds matching KB runbooks; Investigation owns diagnosis. Passing a pre-formed hypothesis to Investigation anchors it before it has read the logs. The `notes_for_next_stage` field is timeline facts only — observable signals, no causal claims.

![Triage stage YAML structured output: severity, class, KB match and decision](/static/images/agent-tower-notion-hitl-interface/triage-yaml.jpg)

### Two patterns, one mechanism

Both HITL patterns work by detecting a new comment via `_poll_for_new_comment`, then calling `_build_prompt` with `operator_note` appended. The STAGES config makes pattern assignment explicit:

```python
STAGES: list[dict] = [
    {"name": "Triage",        "hitl": None,       ...},
    {"name": "Investigation", "hitl": "pattern_b", "hitl_window_s": 20.0, ...},
    {"name": "Remediation",   "hitl": None,       ...},
    {"name": "Comms",         "hitl": "pattern_a", ...},
]
```

Triage and Remediation have no HITL window. Investigation gets Pattern B (time-bounded). Comms gets Pattern A (indefinite gate with keyword approval).

### Phase-reset: one stage, not the whole run

When the operator leaves a note, `pipeline.py` re-runs only the affected stage. Prior outputs are preserved and passed in as `PRIOR STAGE OUTPUT(S)`. Triage's KB retrieval and Investigation's log analysis do not re-run.

```python
# Pattern B: Investigation — poll after completion
if hitl == "pattern_b" and not dry_run:
    _post_comment(card_id, trimmed, label=name)
    known_comment_ids = _comment_ids(_get_comments(card_id))
    comment = _poll_for_new_comment(card_id, known_comment_ids, timeout_s=window_s)
    if comment:
        prompt_rev = _build_prompt(
            stage, incident_text, log_text, prior_outputs, operator_note=comment
        )
        output = _run_hermes(prompt_rev, run_dir, f"{name}-revised", dry_run)
```

`_build_prompt` appends the operator note at the end of the assembled prompt, after prior stage outputs and the incident report. The agent sees the full context plus the correction. State from earlier stages is preserved; the correction costs one stage.

### Operator notes as durable memory

Pattern A introduces a subtlety: the operator's formatting instruction from Run 1 should apply to Run 2 automatically, without re-stating it. That is the difference between a demo and a system.

The Comms prompt handles this in Step 1. Before drafting, the Comms agent searches past Hermes sessions via FTS5 for prior operator corrections:

```
BEFORE drafting, search past Comms sessions for prior operator corrections.
Use session search with these queries:
- "Applied operator note"
- "operator note 3 bullets customer impact"
```

When a prior correction is found, the agent opens its response with a marker:

```
[Applied prior guidance: 3 bullets, lead with customer impact — recalled from prior session]
```

This marker is the contract. Memory only earns its keep when the prompt actively asks for it and the retrieval result is checkable. If the Comms agent silently changed its output style between runs, there would be no way to know whether memory fired or whether the model just happened to produce a different format.

The three-layer memory architecture makes this possible:

![Three-layer memory: Knowledge (Confluence KB), Operational (Hermes FTS5), Workflow (Notion kanban)](/static/images/agent-tower-notion-hitl-interface/three-layer-memory.svg)

Confluence holds static knowledge: KB articles created by the Doco Agent from past Slack threads. Hermes FTS5 holds operational memory: per-session outputs that agents can search across. Notion holds workflow state: card status, agent outputs as comments, operator corrections. Conflating any two of these layers is the common mistake. The Comms agent querying Confluence for formatting guidance, or Triage querying Hermes sessions for KB articles, would produce noise. Each layer has a distinct access pattern.

## Results

**Run 1**: the pipeline runs all four stages. Triage matches a connection-pool KB runbook (medium confidence) and routes to Investigation. Investigation posts findings; the operator catches a wrong hypothesis — network was the assumed cause — and leaves a comment redirecting it to disk I/O. Investigation re-runs with the correction. Comms drafts a verbose brief; the operator leaves "use 3 bullets, lead with customer impact." Comms re-runs and applies the format.

## What Was Hard

**HITL window timing.** Twenty seconds for Pattern B felt tight during demos. Too short and the operator misses it; too long and the audience watches a progress indicator. There is no principled answer — 20 seconds was the best tradeoff for a 30-minute talk. A production system would make this configurable per incident class.

**Hermes echoes the full prompt.** Each stage's stdout starts with the entire prompt text, including prior stage outputs, before the agent response. Extracting the NOTION summary line and the response body required finding the last "Initializing agent..." marker in stdout and scanning only what follows. Naive substring search returns the prior stage's NOTION line, not the current one.

**FTS5 retrieval finds the prompt itself.** The Comms prompt contains the search query strings it will run. On first execution those same terms appear in the session that was just created. The prompt had to explicitly instruct the agent to ignore hits that contain query strings but no `[Applied operator note: ...]` marker. Without that instruction, the agent retrieves its own prompt as prior guidance.

**Phase-reset from mid-pipeline.** `--from-stage Investigation` is supported for demo re-runs without re-running Triage. `pipeline.py` reads prior stage outputs from `.run/` on disk to reconstruct prior context. Getting file-naming conventions consistent across live runs, dry runs and fixture replays was the majority of the debugging time.

## Where It Goes Next

**Mem0 for operational memory.** Hermes FTS5 works for exact-phrase search but struggles when the prior operator correction was phrased differently from the search query. Mem0 or a vector store would handle semantic retrieval without rewriting the pipeline structure.

**Langfuse or LangSmith for observability.** The current build has no tracing layer. What did the agent decide? That lives only in Notion comments. A proper observability layer would surface stage outputs, token counts and latency without opening the terminal. This is the sharpest gap between demo and production.

**Mixed human/agent board.** The natural extension is not a technical upgrade; it is a conceptual one. If a kanban card can represent an agent run, it can equally represent a human task. Both post comments, both steer the pipeline, both show as status changes on the same board. The surface for mixed human/agent coordination already exists.

![Future-state mixed kanban: humans and agents coordinating on the same board surface](/static/images/agent-tower-notion-hitl-interface/mixed-board.svg)

This is the direction Notion 3.3 (February 2026) moved when it shipped native agent orchestration. Reusing the collaboration layer rather than building a control UI turned out to be where the platform went. Reading that six months early was the validation.

## Code and Further Reading

The full source is on [GitHub](https://github.com/cchew/agent-tower). The stack is Python 3.12, [Hermes Agent](https://github.com/NousResearch/hermes) (Nous Research), Notion REST API and httpx.

For observability, [Langfuse](https://langfuse.com) is the most straightforward drop-in for Python-based agent pipelines.

For replacing FTS5 memory with semantic retrieval, [Mem0](https://mem0.ai) integrates with most Python agent frameworks without changing the pipeline structure.
